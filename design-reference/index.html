B ── Why a swarm

Decentralized
by design.

Most "AI coding" tools rely on a single supervisor that becomes the bottleneck the moment your project gets interesting. Chorus runs peers — like an ant colony — that self-organize, vote, and recover from each other's failures.

True peer-to-peer mesh

Agents communicate via a shared blackboard (Redis pub/sub) and dynamically hand off work to whichever peer is least busy. No central controller. No single point of failure.

root ↔ backend ↔ frontend GOSSIP

Plan, then execute

Plan Mode generates a SPEC.md with diagrams. Edit it inline. Approve. The swarm builds exactly what you signed off on.

Time-travel checkpoints

Every file write is a checkpoint. Rewind the whole swarm to any moment and fork a new branch from there.

Direct any agent

Pause, resume, or whisper a directive to a single agent without halting the rest of the swarm.

Sandbox-first execution

Seatbelt on macOS, bubblewrap on Linux. 100ms cold start, no Docker daemon required.

RAG-grounded code

pgvector-backed knowledge base means agents cite real docs, not hallucinated APIs.

Ship a ZIP. Or a running container. Or both.

Every Chorus project graduates with a verified ZIP archive and a built Docker image. The PackagerAgent runs last, verifies every file checksum, and produces both deliverables in parallel.

C ── The flow

Four steps,
one outcome.

From a one-line idea to a production codebase you can clone, run, and extend.

01 / DESCRIBE

Tell Chorus what to build

"A task manager with JWT auth, Postgres, and an admin dashboard." That's enough.

02 / PLAN

Review the SPEC.md

Agents draft an architecture diagram, schema, and module list. Edit anything.

03 / BUILD

Watch the swarm work

Six agents claim files in parallel. Pause, fork, or whisper directives at any moment.

04 / SHIP

Download or deploy

ZIP archive + Docker image, both verified. Or push to GitHub with one click.
